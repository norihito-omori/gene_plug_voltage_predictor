# baseline+16 越境定義 × 稼働率正規化シグナル × タイムリーアラート最適化

## 1. 目的 (Why)

exp-012 では越境基準を固定 32kV として評価した。本実験では以下の 2 点を変更し、より実用的なアラートルールを探索する。

1. **越境定義の変更**: `daily_max >= 32kV` → `daily_max >= baseline + 16`
   - `baseline` は gen_no ごとの稼働開始 30 日間の `daily_max` 中央値
   - プラグ個性（ゲタ）を吸収し、「個体ごとの正常値から 16kV 上昇」という条件で統一
   - 「より激しく劣化した事例」に絞り込むことで、精度の高いアラートを設計する

2. **シグナルの変更**: `hours_at_31kV`（絶対閾値）→ `rate_at_(baseline+N)` = 稼働時間正規化レート
   - `rate_at_(baseline+N)` = (要求電圧 == round(baseline) + N の稼働スロット数 × 0.5h) / 当日稼働時間
   - 連続運転・断続運転の機台を同一基準で比較可能
   - 当日稼働時間 < 1.0h の日は NaN（評価対象外）

3. **最適化指標の変更**: recall 最大 → **timely_rate 最大 → too_early 最小 → missed 最小**
   - 「早すぎる交換はコスト損害」という業務制約を反映
   - 越境の 7〜14 日前に初回アラートが出る割合（timely_rate）を最優先で最大化

## 2. データ・条件 (Setup)

| 項目 | EP370G | EP400G |
|-----|--------|--------|
| クリーニング済み CSV | `cleaned_ep370g_v5.csv` | `cleaned_ep400g_v3.csv` |
| 日次データセット | `dataset_ep370g_ts_v7.csv` | `dataset_ep400g_ts_v3.csv` |
| baseline+16 越境イベント数 | **44件** | **14件** |
| 非越境サイクル数 | 123件 | 131件 |
| 比較: 固定 32kV 越境イベント数 (exp-012) | 113件 | 28件 |

EP400G v3 の変更点: 4 機場（8200/8930/8980/9040）に `EP400G_START_DATETIMES` カットオフを追加。baseline=2kV の異常期間データを除外し、最低 baseline が 2kV → 16kV に改善。

### 越境イベント数が減った理由

「32kV 越境」より「baseline+16 越境」のほうがイベント数が少ない。これはプラグの baseline が 16kV 未満の場合、`baseline + 16 < 32kV` となり、絶対基準では越境でも相対基準では越境に含まれないケースが多いため。**本実験の対象は「真に激しく劣化した事例」に絞られている。**

### シグナル定義

```
Lv1: rate_at_(baseline+13)  ← 越境の 3kV 手前
Lv2: rate_at_(baseline+14)  ← 越境の 2kV 手前
Lv3: rate_at_(baseline+15)  ← 越境の 1kV 手前

アラートシグナル: rate の W 日ローリング平均 > threshold
```

グリッドサーチ範囲:
- W（ウィンドウ）: [3, 5, 7, 10, 14, 21 日]
- threshold: [0.01, 0.05, 0.10, 0.20, 0.30, 0.50]

タイムリー定義: T_MIN=7, T_MAX=14（越境の 7〜14 日前に初回アラート）

## 4. 考察 (Discussion)

### 4-1. 相対閾値の効果

`rate_at_(baseline+N)` は稼働時間正規化により、連続運転・断続運転の機台を公平に評価できる。ただし、以下の構造的制約がある。

- **Lv1 (bl+13)**: タイムリー率が最も高いが too_early も多い。3kV 手前のシグナルは先行日数が長くなる（EP370G med=14日）一方、誤発動しやすい
- **Lv2 (bl+14)**: タイムリー率と too_early のバランスが最良。EP400G では too_early=0 を達成
- **Lv3 (bl+15)**: 越境 1kV 手前のため先行日数が短く、timely% の改善に貢献しない。recall は高いが too_early が激増

### 4-2. EP370G と EP400G の性質の違い

EP370G は 44件のイベントで機台数が多く、タイムリー率の最大値でも 40.9%（18/44件）に留まる。EP400G は 14件と少なく、too_early=0 を達成できるが recall=64.3%（9/14件の見逃し）という制約がある。

EP400G で too_late が多い（5/9 検知件の初回アラートが越境 7 日以内）のは、EP400G が EP370G より劣化進行が速く、bl+14 に到達してから越境までのリードタイムが短い事例が多いため。

### 4-3. 多段アラート設計の可能性

`optimize_relative_alert.py` の多段評価結果（各レベルのベストパラメータで OR 条件）から、初回発火レベルの分布を確認。EP370G では Lv1 が先に発火するケースが圧倒的多数（構造的に bl+13 → bl+14 → bl+15 の順に発火する）。

多段アラートとして設計する場合:
- **Lv1 (bl+13, w=3d, thr=0.30)**: 初期警告（「要観察」）
- **Lv2 (bl+14, w=3d, thr=0.30)**: 交換部品の手配推奨
- **Lv3 (bl+15, w=3d, thr=0.10)**: 緊急対応

ただし、各レベル単体での timely_rate が 40% 前後に留まることから、多段化による精度向上は限定的。見逃し（missed）の構造的原因の解消が先決である。

### 4-4. 見逃しの原因

EP370G missed=5件・EP400G missed=5件は、越境前の期間に `rate_at_(baseline+13/14/15)` がいずれも閾値を超えない。これは bl+13 滞在が少ない（稼働時間に対する率が低い）ことに起因し、センサー分解能 1kV の構造的限界と同一の根本原因。

## 5. 推奨アラートルール (結論)

評価方針: **timely_rate 最大 → too_early 最小 → missed 最小**（早すぎる交換 = コスト損害、1 週間前アラートが最適）

| 機種 | レベル | アラート条件 | Recall | timely% | too_early | FP率 | med_lead |
|-----|--------|------------|--------|---------|-----------|------|---------|
| **EP370G** | Lv2 | `rate_at_(bl+14) の 3日 rolling mean > 0.30` | 86.4% (38/44) | **38.6%** (17/44) | 14 | 17.9% | 10d |
| **EP400G** | Lv2 | `rate_at_(bl+14) の 3日 rolling mean > 0.30` | 64.3% (9/14) | **28.6%** (4/14) | 0 | 3.8% | 3d |

- 両機種で **Lv2 (baseline+14)、3日ウィンドウ、0.30 閾値** が最適
- EP400G では too_early=0（早すぎる交換ゼロ）を達成
- EP370G の too_early=14 は Lv1 (bl+13) と比較して 4件削減（18→14）、FP率も 25.2%→17.9% に改善
- **限界**: EP370G timely=17/44（57% が範囲外アラートまたは見逃し）、EP400G timely=4/14（71% が同）は業務要件との整合確認が必要

### exp-012 との比較

| 指標 | exp-012 (固定 32kV, recall 最大) | exp-013 (baseline+16, timely 最大) |
|-----|--------------------------------|----------------------------------|
| EP370G Recall | 91.2% (103/113) | 86.4% (38/44) ※対象イベント数が異なる |
| EP370G timely% | 未評価 | 38.6% |
